## Architect review — session diff 80eac87..61144b4

### Top 5 structural risks (highest impact first)
1. Response cache is not scoped to the active route/workload epoch, so it can return stale outputs after model swaps.
   - File: packages/core/src/openaiProxy.ts:592
   - Problem: Response cache identity is currently `{sha, model}` (`model` comes from request body, `sha` from canonical request JSON) and lookup does not include route identity, workload epoch, or model artifact identity before serving a hit (`maybeResponseCacheLookup` + early return path). If a model alias is re-pointed to a different workload (or the same workload changes epoch), deterministic requests can continue returning responses generated by the old backing runtime, bypassing both routing and KV validation. This is a correctness risk, not just a hit-rate issue.
   - Fix: Extend response-cache key space to include route-scoped provenance at lookup/persist time: at minimum `{model, workload, workloadEpoch}` and ideally model artifact fingerprint. Resolve route first, attach route metadata to `ResponseCacheRequestState`, persist it in `response_entries`, and invalidate old rows on epoch change. This keeps deterministic replay bounded to the exact runtime that generated it.

2. Anthropic responses are cached pre-translation and replayed through a live translator, coupling cache correctness to translator version drift.
   - File: packages/core/src/openaiProxy.ts:1208
   - Problem: The proxy persists response-cache entries before anthropic translation (`maybePersistResponseCache` runs on upstream OpenAI payload), then translates on read (`maybeTranslateResponse` on cache hit). That means persisted bytes are protocol-internal, while user-visible bytes are produced by whatever translation code is loaded at replay time. A translator behavior change can silently alter cached anthropic outputs without cache invalidation boundaries, which makes behavior non-repeatable across deploys.
   - Fix: Split cache domains by external API surface and translator version. Either cache post-translation payloads for `/v1/messages` (preferred for API determinism), or embed translator version in cache metadata/key and invalidate incompatible rows on startup. Keep OpenAI and Anthropic cacheability independent so protocol adapters are not hidden cache dependencies.

3. KV and response cache use different request canonicalization strategies, creating incoherent warm paths and duplicate state.
   - File: packages/core/src/openaiProxy.ts:743
   - Problem: KV identity uses raw `sha1(bodyText)` while response cache uses canonicalized JSON hashing (`canonicalRequestSha`). Equivalent deterministic requests with different key ordering can hit response cache and miss KV (or create distinct KV entries), so two cache layers disagree on identity for the same semantic request. This undermines predictable layering and makes hit/miss behavior hard to reason about.
   - Fix: Define one shared request-identity contract in `packages/core/src/cache-identity/` and consume it from both KV and response cache. Use canonicalized request hashing plus explicit per-layer discriminators (e.g., workload epoch for KV, protocol variant for response cache) so both layers agree on semantic identity while preserving layer-specific safety checks.

4. Slot allocation is process-local, single-slot-by-default, and drop-on-contention; concurrent traffic degrades cache capture and replay determinism.
   - File: packages/core/src/kvstore/slotAllocator.ts:23
   - Problem: `SlotAllocator(1)` is per-process/per-workload and non-blocking (`acquire()` returns `null` when busy). In `maybeKvLookup` and `maybePersistKv`, contention immediately disables restore/save paths rather than queueing or retrying, so hot workloads under parallel load can oscillate between warm and cold behavior and fail to persist new reusable slots. There is also no inter-process coordination if multiple proxy processes share the same runtime root.
   - Fix: Move slot coordination to a durable lease mechanism in `kvstore` (SQLite-backed lease table with TTL/owner) and add bounded wait/backoff semantics for restore/save operations. Keep a fast in-process path for uncontended cases, but make contention behavior explicit and observable instead of silent null-acquire fallback.

5. `useProxy` consumption is hard-coded around `ModelRun`, leaving no extensible seam for `ModelHost` routing policy evolution.
   - File: packages/cli/src/commands/supervisor.ts:346
   - Problem: Supervisor startup rewriting explicitly skips non-`ModelRun` workloads, and composite apply computes `workloadEndpoint` from `spec.useProxy` in the ModelRun synthesis path only. This duplicates policy in call sites and bakes a kind-specific assumption into runtime routing, making future `ModelHost` parity or shared routing policy harder and error-prone.
   - Fix: Promote proxy-routing resolution into a shared remote/core policy function keyed by resolved workload runtime metadata (kind-agnostic), and call it from supervisor/composite/apply surfaces. Treat `useProxy` as one input into route policy, not as inline branching in each consumer.
