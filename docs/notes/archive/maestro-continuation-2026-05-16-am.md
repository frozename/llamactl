## Three real bugs surfaced this session (worth knowing)

1. **Zombie llama-server zero-effect bug.** During an inline quick eval, I observed adapter and base producing byte-identical outputs. Cause: both runs hit a leftover llama-server on :18099 (the prior train-lora.sh smoke server didn't get killed cleanly). New servers failed to bind, exited silently; curl-to-health responded from the zombie. The new `eval-classifier.sh` guards against this by hitting `/props` after server start and verifying the loaded model path. The spike's smoke step also has this risk and doesn't yet have the same guard — worth porting back.

2. **n_predict=150 caps the eval signal.** Test prompts ask for `{"memory_related": bool, "reason": "..."}`. With temperature 0 the model fills the `reason` field with prose that uses up the 150 tokens before the closing brace. Result: most responses fail JSON parse despite being shape-correct. Bumping to ~250-300 should triple parse rate. The eval driver's `n_predict` is hardcoded — make it a param or just bump it in the script.

3. **Memory-efficacy 4-way valid/test sets have ZERO minority-class rows.** Mod-10 stratified split with class sizes 2/4/8 puts every minority row in train. The 4-way framing is process-validation-only by construction; the binary framing carries the real signal but with only 1 positive in valid + 1 in test.
