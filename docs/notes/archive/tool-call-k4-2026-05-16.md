# K.4 — Eval Qwen3-4B-Instruct-2507 +/- LoRA on adversarial-v0

`git log -1 --oneline` at run time: `e84bda3 docs(notes): memory-efficacy M.7+M.8 — retrained adapter is byte-identical`

Test set: `packages/train/corpora/tool-call-grammar/adversarial-v0/seed.jsonl` (n=25, 4 categories) from K.3.

Adapter: `packages/train/.spike-work/tool-call-grammar-qwen3-4b/gguf/adapter.gguf` (trained in K.1 on the easy gold corpus).

Eval: `eval-tool-calls.sh` with `tool_choice` default added (the new corpus didn't carry the `tool_choice` field).

## Results

| | parse success | positives | negatives |
|---|---|---|---|
| base    | 6/25 (24%) | 23 | 2 |
| adapter | 6/25 (24%) | 23 | 2 |

The K.1 adapter does not change parse success on the harder set. K.1 conclusion holds in K.4: **at this rank/layers/iters, the adapter has no measurable lift over base.**

## Per-category breakdown

|category | success (base) | success (adapter) | name-match (both) | failure mode |
|---|---|---|---|---|
| ambiguous-intent | 4/7 | 4/7 | 5/7 | model called when reply expected (or vice versa) |
| multi-tool       | 0/5 | 0/5 | 5/5 | model emits 1st tool only; scorer expects parallel emission |
| name-collision   | 1/5 | 1/5 | 4/5 | right tool, wrong args |
| schema-edge      | 1/8 | 1/8 | 7-8/8 | argument-shape mismatches (datetime regex etc.) |

**Name-match rate: 88-96% across both runs.** The model picks the right tool nearly always; the parse-success metric fails because:

1. **Multi-tool rows (5/25 of the corpus)** assume parallel tool emission. Real OpenAI-style tool use is sequential: call → tool result → next call. The Qwen3 jinja template emits the first call and waits. The K.3 corpus design needs to be revisited — either split multi-tool rows into per-step gold, or change the scorer to treat partial-prefix matches as success.

2. **Schema-edge rows (8/25)** test argument-shape failures (datetime regex, optional field handling). Most fail on argument canonicalization, not tool selection. This is real adapter-trainable signal, but the K.1 adapter wasn't trained on these patterns so it doesn't help.

3. **Ambiguous-intent rows (4/7 success)** are the only category where the metric is meaningful out-of-the-box — the model either calls or replies, and you can score that cleanly.

## Sample mismatch (adv-001, multi-tool)

Gold expects 2 calls: `weather_current({"location":"Paris"})` then `unit_convert(...)` for the 20°C→°F conversion.

Model emits 1 call: `weather_current({"location":"Paris"})` and stops, expecting a tool result before deciding the next call.

This is correct tool-call behavior for sequential turn-based dispatch. The scorer reports `len(tool_calls) == len(gold_calls)` → False → fail.

## Verdict

K.4 exposes two truths:

1. **The K.3 corpus is doing its job** — 88-96% name-match rate hides 24% parse-success. The harder set surfaces failure modes the easy gold doesn't.
2. **The current scorer is too strict for multi-tool rows.** Two options going forward: (a) split each multi-tool row into a sequence of single-turn gold pairs with synthetic tool results, or (b) extend the scorer to accept a prefix-match for sequential tool emission.

Adapter-track conclusion (K.1+K.2+K.4 combined): **The K.1 adapter does not lift parse success at any difficulty level.** Either (a) the rank=16 LoRA capacity is too small to encode the failure-mode patterns, or (b) the K.1 training corpus (easy gold) doesn't expose the model to the patterns that fail in adversarial-v0. Worth trying K.5: train on adversarial-v0 directly with adapter-friendly framing (single-turn per row) and re-eval on a held-out half.
