# Agentic-eval framework — design

Date: 2026-05-06
Status: draft, awaiting review

## Background

Local llama.cpp models on the M4 Pro (`local`, 48 GB) and the mac-mini
(`mac-mini`, 16 GB) get used as penumbra agents — `local-gemma26`,
`local-qwen36-27b`, `local-qwen35-4b`, etc. Today there is no objective,
reproducible signal for how good any of those models actually is at the
specific shapes penumbra demands: tool-calling against MCP schemas,
JSON-structured outputs, retrieval from long context, and raw throughput
under sane Apple Silicon tuning.

The MTP pilot (`docs/superpowers/specs/2026-05-05-llamacpp-mtp-pilot-design.md`,
results in `2026-05-05-llamacpp-mtp-pilot-slice-a-results.md`) produced a
working `bench-client.py` that hits `llama-server`'s `/completion` for
9 prompts and reports decode tps. That harness is too thin for agent-fitness
evaluation: no tool-calling, no structured-output check, no retrieval test,
single tuning configuration. The MTP run also used suboptimal Apple Silicon
flags (no `-ub`, no `--flash-attn`, no `-ngl 999`) — a finding that came
from a community tip after the fact.

This spec extends the bench tooling into a proper agentic-eval framework
that produces per-model report cards and a leaderboard surface, so we can
make data-driven decisions about which models to deploy for which penumbra
roles.

## Goal

Build an objective, reproducible bench framework that, for each candidate
local model on each compatible Apple Silicon node, produces:

1. A per-model markdown report card under
   `docs/superpowers/specs/<date>-model-eval-<id>.md`.
2. An aggregated leaderboard surfaced via a new MCP tool
   `llamactl_models_leaderboard`.

Phase 1 is "tight": objective metrics only (no LLM-as-judge), four
sub-benches (throughput, tool-calling correctness, context retrieval,
JSON output validity), one tuning sweep dimension (`-ub` ∈ {256, 512}).

Phase 2 (out of this spec) layers on role-specific scoring (planner /
executor / tester / reviewer) using LLM-as-judge if Phase 1 data
justifies the added cost.

Penumbra auto-routing integration (using eval scores in `agent_recommend`)
is explicitly deferred until both phases land and prove informative.

## Non-goals

- No LLM-as-judge in Phase 1. All scoring is objective / countable.
- No penumbra dispatch path in Phase 1. Bench hits `llama-server` directly
  via `/v1/chat/completions`. (Penumbra-shaped tool _schemas_ are used in
  fixtures; the _dispatch_ is direct.)
- No model fine-tuning, no quantization work. Bench uses GGUFs as
  downloaded.
- No remote / cloud model evaluation. Local llama.cpp only.
- No automatic model discovery. Portfolio is human-curated and pinned in
  the eval config.
- No `agent_recommend` integration in Phase 1.

## Artifact shape

### Per-model report card

Path: `docs/superpowers/specs/<YYYY-MM-DD>-model-eval-<model-id>.md`

Generated by `packages/eval/src/report/render-card.ts` from the JSON
artifacts under `$DEV_STORAGE/eval/`. Sections:

- Model identity (id, GGUF path, file size, source HF repo, commit/SHA
  if available)
- Hardware matrix: results from each node where the model fits
- Per sub-bench: score, supporting metrics, notable failures
- Tuning sweep: `-ub 256` vs `-ub 512` deltas
- Verdict line: a one-sentence "good for X / weak at Y / not recommended"

### Leaderboard MCP surface

Tool: `llamactl_models_leaderboard`

Returns a sortable matrix:

```typescript
type LeaderboardRow = {
  model: string; // catalog id
  node: string; // 'local' | 'mac-mini'
  ub: 256 | 512; // tuning config (best of the two)
  throughput_tps: number; // decode tps mean
  ttft_ms: number; // time to first token
  tool_call_score: number; // [0, 1]
  context_8k_score: number; // [0, 1]
  context_16k_score: number | null; // null if model can't fit ctx
  json_score: number; // [0, 1]
  composite: number; // weighted average, see scoring below
  asof: string; // ISO timestamp of latest run
};
```

Inputs: optional `node`, `min_throughput`, `min_tool_call_score`,
`sort_by` filter/sort knobs. Default sort: descending `composite`.

The MCP tool reads from a small SQLite store at
`$DEV_STORAGE/eval/leaderboard.sqlite` populated by the eval runner.

The existing `tools/llama-cpp-mtp/` stays untouched. It remains the
reference for MTP-specific work and gets re-run as a side-quest under
this spec's tuning flags.

### Server lifecycle

Each eval run for a `(model, ub)` pair:

1. Resolve GGUF path from catalog `rel`.
2. Spawn `llama-server` on a non-default port (e.g. `18181`) with pinned
   args:
   ```
   --host 127.0.0.1 --port 18181 --model <gguf>
   --ctx-size 8192 --no-warmup -np 1
   --n-gpu-layers 999 --flash-attn -ub <ub>
   ```
3. Poll `/health` until 200 (timeout 120s).
4. Run all four sub-benches against the server.
5. Kill server, persist results to `$DEV_STORAGE/eval/<run-ts>/<model>-<node>-ub<ub>.json`.
6. Update SQLite leaderboard.

Catch: each `(model, ub)` pair is a fresh server boot. Eight evals per
model on M4 Pro (4 sub-benches × 2 ub) but only 2 server boots. Boot is
~10s; total bench time per model dominated by inference, not boot.

### Tuning sweep

Pinned across all runs:

- `-ngl 999` (all-on-GPU; sane Apple Silicon default)
- `--flash-attn` (sane default per community wisdom)
- `-c 8192` (sane ctx; long-context sub-bench expands per-prompt)
- `-np 1` (single-user; no concurrency noise)

Swept: `-ub ∈ {256, 512}`.

Spot-validation: one model (Qwen3.6-27B Q4_K_M) gets a third config with
`--flash-attn off, -ub 512` to verify the recommendation. If `flash-attn off`
shows >5% deviation in either direction, expand the sweep on the next
revision.

### MTP re-bench (side-quest)

The 2026-05-05 MTP pilot ran with no `-ub`, no `-ngl`, no `--flash-attn`.
Under the spec's new tuning flags, MTP may or may not flip from 0.85x to
something interesting. This spec adds one task: re-run vanilla and MTP
benches on `qwen36-27b-q4m` and `qwen36-27b-mtp` under `-ngl 999
--flash-attn -c 8192 -ub 512`. If MTP flips above the original 1.4x gate,
the MTP pilot's "abandon Slice B" verdict is reopened.

## Sub-bench specifications

### 1. Throughput

Same shape as `tools/llama-cpp-mtp/bench-client.py`. Nine fixed prompts
covering code generation, explanation, summarization, QA, translation,
creative, math. Each prompt: `n_predict=192`, `temperature=0`, `seed=42`,
`cache_prompt=false`, non-streaming.

Captures from server timings:

- `predicted_per_second` per prompt
- `prompt_per_second` per prompt
- TTFT (synthesized from wall_time - predicted_n / predicted_per_second)
- Aggregate decode tps mean + p10 + p90

Score: throughput_tps = aggregate mean decode tps. Reported as raw number,
not normalized.

### 2. Tool-calling correctness

Fixture file: `prompts-tool-calling.json`. ~12 prompts, each paired with
the same penumbra-shaped tool schema set in `tools-penumbra.json`. Gold
answer per prompt is one of:

- `{ should_call: false }` — model should NOT call any tool
- `{ should_call: true, tool: "name", args_predicate: { ... } }` — model
  should call this specific tool, with args matching the predicate

Tools (penumbra-shaped):

- `chain_start(initial_agent: string, message: string, task_type?: enum)`
- `handoff_approve(handoff_id: string)`
- `task_get(task_id: string)`
- `memory_search(query: string, project_id?: string, limit?: integer)`
- `fs_grep(pattern: string, path?: string)`

Example fixtures:

- "Find references to `LLAMA_CPP_BIN_MTP` in the codebase" → fs_grep
- "What did we decide about MTP gate criteria?" → memory_search
- "Approve handoff 8337e3c8-bef1-4f55-8326-89879833ac6d" → handoff_approve
- "Write a haiku about the moon" → no tool call
- "Get me the status of task 5" → task_get with `task_id: "5"`

Per-prompt scoring (binary):

- `valid_json`: did the model emit a parseable `tool_calls` array?
- `correct_decision`: did `should_call` match?
- `correct_tool`: if should_call, did `tool` match?
- `args_match`: if should_call, do args satisfy the predicate?

Aggregate score: `tool_call_score = mean(valid_json AND correct_decision
AND (correct_tool if should_call else true) AND (args_match if should_call
else true))` ∈ [0, 1].

### 3. Context retrieval (needle-in-haystack)

Fixture file: `prompts-context.json`. Three depths: 4k, 8k, 16k tokens.
For each depth: assemble a haystack from a deterministic Wikipedia-style
filler (committed to repo as `fixtures/haystack-base.txt`), insert one
"needle" sentence at depth 0.25, 0.5, 0.75 (3 positions × 3 depths = 9
runs).

Needle examples:

- "The quartermaster of the HMS Endeavour in 1771 was named Charles
  Whittington."
- "The first commit hash recorded in the experimental ledger was
  `b1c4a7f31d`."
- (etc — 9 unique needles)

Question after the haystack: "What was the [quartermaster of the HMS
Endeavour in 1771]?" / "What was [the first commit hash in the
experimental ledger]?"

Per-prompt scoring (binary): exact substring of the needle appears in
the model's response.

Aggregate score: `context_<depth>_score = mean(found)` per depth.

If `-c 8192` doesn't fit the 16k haystack, the runner sets the model's
ctx to `<depth + 512>` for that sub-bench specifically (override, not
sweep).

### 4. JSON output structure validity

Fixture file: `prompts-json-output.json`. Five prompts asking for
structured JSON output. Each fixture pairs a prompt with a JSON Schema.

Examples:

- "Extract the entities mentioned in this text and return as JSON
  matching `{entities: [{name: string, type: 'person'|'place'|'org'}]}`"
- "Parse this changelog into a JSON list of `{version, date, breaking,
added, fixed}`"

Per-prompt scoring (binary): does the response contain a JSON block that
parses AND validates against the schema?

Aggregate score: `json_score = mean(valid)` ∈ [0, 1].

### Composite score

```
composite = 0.30 * normalized_throughput
          + 0.30 * tool_call_score
          + 0.20 * context_8k_score
          + 0.10 * (context_16k_score ?? context_8k_score)
          + 0.10 * json_score
```

`normalized_throughput` is `min(1.0, throughput_tps / 30)` — 30 tps is the
target ceiling; faster models cap at 1.0 to avoid speed dominating quality.

The weights are deliberately conservative. Tool-calling and throughput are
tied at 30% because penumbra agent usefulness depends on both. Context
retrieval gets 30% combined because long-context handling is the
differentiator from 4B-class models. JSON output is 10% because it's a
prerequisite, not a differentiator.

Weights live in `packages/eval/src/score/compose.ts` and are easy to
revisit after Phase 1 data.

| Node       | Profile                  |   RAM | Models in scope                                                                                                                                                                                                        |
| ---------- | ------------------------ | ----: | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `local`    | macbook-pro-48g (M4 Pro) | 48 GB | All M4 Pro tier (gpt-oss-20b, Mistral 24B, Granite-4-Code-20B, QwQ-32B-Preview) + all baselines that fit (gemma31, gemma26, qwen36-35b-moe, qwen36-27b, qwen35-27b, qwen3-coder) + all mac-mini tier (cross-validates) |
| `mac-mini` | mac-mini-16g             | 16 GB | Mac-mini tier only (Qwen3.6-8B, Llama-3.3-8B, Phi-4-mini, Granite-4-3B-Code) + qwen35-4b baseline                                                                                                                      |

Auto-skip rule: if model GGUF size > (node_ram_gb \* 0.6 GiB), skip the
(node, model) pair and record `skipped: oom_likely`.

## Portfolio

Two phases. Both phases run end-to-end (download → bench → report → leaderboard
update). Phase 1 ships first because it has zero new downloads; Phase 2
adds 8 candidates and runs after Phase 1's leaderboard is published and
sanity-checked.

### Phase 1 — baseline existing penumbra agents (no downloads)

| --------------------------------------------- | ----------------- | ------ |
| existing — see `packages/core/src/catalog.ts` | (already on disk) | M4 Pro |

Concrete ids (cross-checked against current penumbra agent list):
`gemma26`, `gemma31`, `qwen36-35b-moe`, `qwen36-27b`, `qwen35-27b`,

### Cost estimate

Per-pair runtime (one bench per `(model, node, ub)`): ~10–15 minutes
including server boot. Throughput sub-bench dominates (9 prompts × ~5s
average = ~45s); tool-calling adds ~2 min (12 prompts, each shorter); context
sub-bench adds ~5 min for 8k haystack alone; JSON output adds ~1 min.

Total runs: ~15 models × 2 ub × 1–2 nodes (per fit) × 1 sub-bench-set ≈
50–60 server boots and ~10–15 hours wall time across both nodes,
parallelisable since each node runs independently.

## Cross-repo impact

`llamactl` only.

The new `llamactl_models_leaderboard` MCP tool is exposed via `packages/mcp/`
and becomes visible to other repos (penumbra, sirius-gateway) as a normal
MCP tool. Penumbra's `agent_recommend` is unaffected — it stays on its
existing telemetry-driven ranking until Phase C explicitly wires the eval
scores in.

No `sirius-gateway` or `embersynth` source changes.

## Reversibility

Fully reversible:

- Delete `packages/eval/` package directory.
- Drop the `llamactl_models_leaderboard` MCP tool.
- Remove the `llamactl eval ...` CLI command.
- Delete `$DEV_STORAGE/eval/` directory (bench JSONs, sqlite store).
- Remove the per-model report cards from `docs/superpowers/specs/`.
- Phase 2 GGUF downloads under `$LLAMA_CPP_MODELS/` can be cleaned up
  with `rm -rf <Phase2 model dirs>`.

No catalog schema changes, no penumbra integration. The only
production-visible surface in Phase 1 is the new MCP tool and CLI command,
both opt-in.

## Open questions

None blocking. Decisions deferred to implementation:

- Exact catalog id naming for new entries — confirmed during plan execution
  against `packages/core/src/catalog.ts` conventions.
- Whether the MCP tool returns one row per `(model, node, ub)` or
  collapses to best-of-ub per `(model, node)` — defaults to collapsed
  with a `--detailed` opt-in flag.
- Where the JSON Schema validator dependency comes from (zod is already
  in tree; reuse it for fixture schemas).
- Reranking weights post-Phase-1 — driven by data, not pre-decided.

## Re-evaluation triggers (Phase 2 / Phase C)

After Phase 1 publishes a leaderboard:

- If ≥3 models score above a composite threshold (TBD post-data), proceed
  to Phase 2 layered role-based scoring with LLM-as-judge.
- If exactly one model dominates, skip Phase 2 and discuss promoting the
  winner to default agent role.
- If no model breaks a useful threshold, the bench has done its job —
  the answer is "wait for better local models" and Phase 2 is deferred.

Penumbra `agent_recommend` integration becomes interesting once Phase 2
data exists; not before.
